# 테이블오더 서비스 - 요구사항 정의서

## Intent Analysis Summary

| 항목 | 내용 |
|---|---|
| **사용자 요청** | 디지털 테이블오더 플랫폼 구축 (고객 주문 + 매장 관리) |
| **요청 유형** | New Project (신규 프로젝트) |
| **범위** | System-wide (프론트엔드 2개 + 백엔드 API + 데이터베이스) |
| **복잡도** | Complex (실시간 통신, 멀티테넌시, 세션 관리, 대규모 운영) |

---

## 1. 기술 스택

| 계층 | 기술 |
|---|---|
| **백엔드** | FastAPI (Python) |
| **프론트엔드** | React + TypeScript |
| **데이터베이스** | PostgreSQL |
| **배포 환경** | AWS (EC2, RDS, S3 등) |
| **실시간 통신 (관리자)** | Server-Sent Events (SSE) |
| **실시간 통신 (고객)** | 폴링 (30초 간격) |

---

## 2. 시스템 구조

### 2.1 멀티테넌시
- **다중 매장 + 본사 관리** 구조
- 각 매장은 독립적 데이터 (메뉴, 주문, 테이블)
- 본사에서 전체 매장 관리 가능

### 2.2 주요 구성 요소
- **고객용 웹 앱**: 태블릿 전용 (매장 내 고정 태블릿), 모바일은 향후 확장
- **관리자용 웹 앱**: 매장 운영자 + 본사 관리자용 관리 화면
- **백엔드 API**: RESTful API 서버 (FastAPI)
- **데이터베이스**: PostgreSQL (AWS RDS)
- **파일 스토리지**: 서버 로컬 파일 시스템 (이미지 업로드)

## 3. 기능 요구사항 (Functional Requirements)

### 3.1 고객용 기능

#### FR-C01: 테이블 태블릿 자동 로그인 및 세션 관리
- 관리자가 1회 초기 설정 (매장 식별자, 테이블 번호, 비밀번호)
- 로그인 정보 로컬 저장 후 자동 로그인
- 16시간 세션 유지

#### FR-C02: 메뉴 조회 및 탐색
- 메뉴 화면이 기본 화면
- 카테고리별 메뉴 분류 및 표시
- 메뉴 상세 정보: 메뉴명, 가격, 설명, 이미지
- 카드 형태 레이아웃, 터치 친화적 (최소 44x44px)

#### FR-C03: 장바구니 관리
- 메뉴 추가/삭제, 수량 조절
- 총 금액 실시간 계산
- 장바구니 비우기
- 클라이언트 측 로컬 저장 (새로고침 시 유지)
- 서버 전송은 주문 확정 시에만

#### FR-C04: 주문 생성
- 주문 내역 최종 확인 후 확정
- 주문 성공: 주문 번호 표시 → 장바구니 비우기 → 5초 후 메뉴 화면 리다이렉트
- 주문 실패: 에러 메시지 표시, 장바구니 유지
- 주문 정보: 매장 ID, 테이블 ID, 메뉴 목록, 총 금액, 세션 ID

#### FR-C05: 주문 내역 조회
- 현재 테이블 세션 주문만 표시
- 주문 시간 순 정렬
- 주문별: 주문 번호, 시각, 메뉴/수량, 금액, 상태(대기중/준비중/완료)
- 30초 간격 폴링으로 상태 업데이트
- 페이지네이션 또는 무한 스크롤

### 3.2 관리자용 기능

#### FR-A01: 매장 인증
- 매장 식별자 + 사용자명 + 비밀번호 입력
- JWT 토큰 기반 인증
- 16시간 세션 유지, 브라우저 새로고침 시 유지
- 비밀번호 bcrypt 해싱
- 로그인 시도 제한

#### FR-A02: 실시간 주문 모니터링
- SSE 기반 실시간 주문 목록 업데이트
- 그리드/대시보드 레이아웃 (테이블별 카드)
- 각 테이블 카드: 총 주문액, 최신 주문 미리보기
- 주문 카드 클릭 시 전체 메뉴 목록 상세 보기
- 주문 상태 변경 (대기중 → 준비중 → 완료)
- 신규 주문 시각적 강조
- 2초 이내 주문 표시
- 테이블별 필터링

#### FR-A03: 테이블 관리
- **초기 설정**: 테이블 번호/비밀번호 설정, 세션 생성
- **주문 삭제**: 확인 팝업 → 즉시 삭제 → 총 주문액 재계산
- **테이블 세션 종료(이용 완료)**: 확인 팝업 → 주문 내역 과거 이력 이동 → 리셋
- **과거 주문 내역 조회**: 테이블별 과거 주문 목록 (시간 역순), 날짜 필터링

#### FR-A04: 메뉴 관리 (MVP 포함)
- 메뉴 조회 (카테고리별)
- 메뉴 등록 (메뉴명, 가격, 설명, 카테고리, 이미지)
- 메뉴 수정
- 메뉴 삭제
- 메뉴 노출 순서 조정
- 필수 필드 검증, 가격 범위 검증

#### FR-A05: 관리자 계정 관리
- 매장별 관리자 계정 등록 기능
- 매장 관리자가 직접 계정 등록 가능

### 3.3 본사 관리 기능

#### FR-H01: 본사 관리자
- 전체 매장 목록 조회 및 관리
- 매장별 운영 현황 모니터링
- (상세 기능은 Application Design 단계에서 정의)

## 4. 비기능 요구사항 (Non-Functional Requirements)

### NFR-01: 성능
- 관리자 주문 모니터링: 2초 이내 신규 주문 표시 (SSE)
- 고객 주문 상태: 30초 간격 폴링
- API 응답 시간: 일반 요청 500ms 이내
- 동시 접속: 500명 이상 지원

### NFR-02: 확장성
- 10개 이상 매장 동시 운영 지원
- 수평 확장 가능한 아키텍처

### NFR-03: 가용성
- 매장 운영 시간 중 서비스 가용성 99.5% 이상

### NFR-04: 사용성
- 터치 친화적 UI (최소 44x44px 터치 타겟)
- 직관적인 카드 기반 레이아웃
- 명확한 시각적 계층 구조

### NFR-05: 데이터 관리
- 주문 이력 영구 보존 (OrderHistory)
- 세션 ID 기반 주문 그룹화
- 클라이언트 장바구니 로컬 저장

## 5. 제외 범위 (Out of Scope)

요구사항 constraints.md 기준:
- 결제 처리 (PG사 연동, 영수증, 환불, 포인트/쿠폰)
- 복잡한 인증 (OAuth, SNS, 2FA)
- 이미지 리사이징/최적화, CMS, 광고
- 알림 시스템 (푸시, SMS, 이메일, 소리/진동)
- 주방 기능 (주방 전달, 재고 관리)
- 고급 기능 (분석, 매출 리포트, 재고, 직원 관리, 예약, 리뷰, 다국어)
- 외부 연동 (배달, POS, 소셜, 지도, 번역)

## 7. 용어 정의

| 용어 | 정의 |
|---|---|
| **테이블 세션** | 고객이 첫 주문 시작 ~ 이용 완료 처리까지의 기간 |
| **이용 완료** | 관리자가 테이블 세션을 종료하는 행위 |
| **과거 이력** | 이용 완료된 세션의 주문 내역 |
| **SSE** | Server-Sent Events, 서버→클라이언트 단방향 실시간 통신 |
| **폴링** | 클라이언트가 주기적으로 서버에 데이터 요청 |

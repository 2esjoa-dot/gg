## 질문

### Q1. 이미지 처리 방식
메뉴 이미지 업로드 시 리사이징/썸네일 처리가 필요한가요?

A) 원본 그대로 저장 (리사이징 없음)
B) 서버에서 리사이징 후 저장 (예: 800x600)
C) 원본 + 썸네일 두 가지 저장
D) AI가 프로젝트에 적합한 방식 결정

[Answer]: A
> **근거**: MVP 단계에서 이미지 처리 복잡도를 줄이고, 5MB 제한으로 충분히 제어 가능. 태블릿 화면에서 원본 이미지를 CSS로 리사이징하면 충분.

### Q2. 메뉴 조회 캐싱
고객 메뉴 조회 시 캐싱 전략이 필요한가요?

A) 캐싱 없음 (매번 DB 조회, MVP 단순화)
B) 인메모리 캐싱 (TTL 기반, 예: 5분)
C) Redis 등 외부 캐시 사용
D) AI가 프로젝트에 적합한 방식 결정

[Answer]: A
> **근거**: 매장당 메뉴 수가 수십~수백 개 수준으로 DB 조회 부하가 낮음. MVP에서 캐싱 레이어를 추가하면 메뉴 수정 시 캐시 무효화 로직이 필요해져 복잡도만 증가. 추후 트래픽 증가 시 인메모리 캐싱 도입 가능.

### Q3. 파일 저장소
이미지 파일 저장 위치를 어떻게 하시겠습니까?

A) 로컬 파일시스템 (uploads/ 디렉토리)
B) AWS S3
C) MVP는 로컬, 추후 S3 마이그레이션
D) AI가 프로젝트에 적합한 방식 결정

[Answer]: C
> **근거**: MVP에서 S3 설정/비용 부담 없이 빠르게 개발. FileService를 인터페이스로 분리해두면 추후 S3 구현체로 교체 용이. 배포 환경(EC2)에서 로컬 저장도 문제없음.

### Q4. 메뉴 데이터 페이지네이션
관리자 메뉴 목록 조회 시 페이지네이션이 필요한가요?

A) 페이지네이션 없음 (전체 조회, 메뉴 수가 적을 것으로 예상)
B) 오프셋 기반 페이지네이션
C) AI가 프로젝트에 적합한 방식 결정

[Answer]: A
> **근거**: 매장당 메뉴 수가 수십~수백 개 수준으로 전체 조회해도 성능 문제 없음. 카테고리별로 이미 그룹핑되어 있어 UI에서 자연스럽게 분류됨. 불필요한 페이지네이션 로직 제거로 개발 속도 향상.

### Q5. 프론트엔드 상태 관리
고객 메뉴 페이지의 상태 관리 라이브러리를 어떻게 하시겠습니까?

A) React 기본 (useState/useEffect만)
B) Zustand
C) React Query (TanStack Query)
D) AI가 프로젝트에 적합한 방식 결정

[Answer]: A
> **근거**: 고객 메뉴 페이지는 단순 조회 화면(카테고리 탭 + 메뉴 리스트). 서버 상태 관리가 단순하고 캐싱 불필요하므로 useState/useEffect로 충분. 외부 라이브러리 의존성을 줄여 번들 크기 최소화. 장바구니는 Unit 4 담당이라 여기서는 조회만 처리.

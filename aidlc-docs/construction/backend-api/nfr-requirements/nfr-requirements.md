# NFR Requirements - 테이블오더 서비스

## 1. 성능 요구사항

| 항목 | 목표 | 측정 방법 |
|---|---|---|
| API 응답 시간 | 일반 요청 500ms 이내 | p95 기준 |
| SSE 주문 전달 | 2초 이내 | 주문 생성 → 관리자 수신 시간 |
| 동시 접속 | 500명 이상 | 부하 테스트 (k6/locust) |
| DB 쿼리 | 단일 쿼리 100ms 이내 | slow query log |

### 구현 전략
- **async/await 전면 사용**: FastAPI + asyncpg로 I/O 블로킹 제거
- **커넥션 풀**: SQLAlchemy async engine pool_size=20, max_overflow=10
- **SSE 인메모리 큐**: asyncio.Queue 기반, 매장별 독립 채널
- **인덱스 최적화**: 자주 조회되는 컬럼에 복합 인덱스 적용 (이미 모델에 정의됨)

---

## 2. 확장성 요구사항

| 항목 | 목표 | 전략 |
|---|---|---|
| 매장 수 | 10개 이상 동시 운영 | 멀티테넌시 (store_id 기반 데이터 격리) |
| 수평 확장 | 서버 인스턴스 추가 가능 | Stateless API 설계 |
| DB 확장 | 읽기 부하 분산 | Read Replica 도입 가능 구조 |

### 구현 전략
- **Stateless 서버**: JWT 토큰 기반 인증, 서버 세션 없음
- **SSE 확장 한계**: 인메모리 큐는 단일 인스턴스에서만 동작. 다중 인스턴스 시 Redis Pub/Sub로 교체 필요 (MVP 이후)
- **DB 파티셔닝**: MVP에서는 불필요. 매장 100개+ 시 store_id 기반 파티셔닝 고려

---

## 3. 가용성 요구사항

| 항목 | 목표 | 전략 |
|---|---|---|
| 서비스 가용성 | 운영 시간 99.5% | 단일 인스턴스 + 헬스체크 |
| 장애 복구 | 5분 이내 | 자동 재시작 (systemd/PM2) |
| 데이터 보존 | 주문 이력 영구 보존 | RDS 자동 백업 |

### 구현 전략
- **헬스체크 엔드포인트**: `/health` (이미 구현됨)
- **프로세스 관리**: systemd 또는 Docker restart policy
- **DB 백업**: RDS 자동 백업 (7일 보존) + 수동 스냅샷
- **SSE 재연결**: 클라이언트 EventSource 기본 재연결 동작 활용

## 4. 보안 요구사항 (MVP 수준)

| 항목 | 구현 |
|---|---|
| 비밀번호 저장 | bcrypt (cost factor 12) |
| 인증 토큰 | JWT HS256, 16시간 만료 |
| 로그인 보호 | 5회 실패 → 15분 잠금 |
| 데이터 격리 | store_id 기반 멀티테넌시 |
| CORS | 허용 origin 제한 (배포 시) |
| HTTPS | Nginx reverse proxy + Let's Encrypt |

## 5. 사용성 요구사항

| 항목 | 목표 |
|---|---|
| 터치 타겟 | 최소 44x44px |
| 주문 완료 피드백 | 5초 표시 후 자동 리다이렉트 |
| 장바구니 유지 | 새로고침 시 로컬 스토리지 복원 |
| 관리자 세션 | 새로고침 시 JWT 유지 |
| 에러 메시지 | 한국어, 구체적 안내 |

---

## 6. 기술 스택별 NFR 매핑

| 기술 | NFR 기여 |
|---|---|
| FastAPI (async) | 성능 — 비동기 I/O, 높은 동시성 |
| asyncpg | 성능 — 네이티브 PostgreSQL async 드라이버 |
| SQLAlchemy 2.0 | 확장성 — async 지원, 커넥션 풀 관리 |
| JWT | 확장성 — Stateless, 수평 확장 용이 |
| PostgreSQL (RDS) | 가용성 — 자동 백업, 페일오버 |
| Nginx | 보안/성능 — HTTPS, 정적 파일 서빙, 리버스 프록시 |
